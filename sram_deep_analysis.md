# SRAM 16MB 深度容量分析：基于盘延迟、队列深度与网络带宽
> [!IMPORTANT]
> 本文档基于 Little's Law（利特尔定律）对 16MB SRAM 进行严格的容量充分性验证。  
> **核心问题**：在零拷贝架构下，IO 数据在 SRAM 中"驻留"的时间由 SSD 延迟决定——SSD 越慢、并发越高，SRAM 中同时驻留的数据就越多。16MB 是否足够？
---
## 一、分析方法论：Little's Law
```
SRAM 占用量 = 吞吐量 × 数据驻留时间
L = λ × W
```
- **L**：任意时刻 SRAM 中同时驻留的数据总量（字节）
- **λ**：数据通过 SRAM 的速率（字节/秒）  
- **W**：单笔 IO 数据在 SRAM 中的驻留时间（秒）
**驻留时间的定义**：
- **写入路径**：数据从网络 DMA 进入 SRAM 开始，到 SSD 写入完成释放 SRAM 缓冲区结束
- **读取路径**：数据从 SSD 读出 DMA 进入 SRAM 开始，到 RDMA 发送完成释放 SRAM 缓冲区结束

## 五、网络带宽反向约束分析
网络是数据的入口/出口，其带宽也约束了 SRAM 的填充速度。

这意味着：即使网络以线速不断涌入数据，只需 100μs 就能填满整个 SRAM DMA 区域。但因为 SSD 端也在持续消费数据（排出），实际不会出现满溢。
### 5.2 稳态下的流量平衡验证
```
网络入口速率:  100 GB/s  (数据进入 SRAM)
SSD 出口速率:   84 GB/s  (顺序读) / 54 GB/s (顺序写)  (数据离开 SRAM)
```
**读取稳态**（SSD→SRAM→网络）：
- SSD 以 84 GB/s 向 SRAM 注入数据
- 网络以 100 GB/s 的能力从 SRAM 消费数据
- 消费速率 > 注入速率 → SRAM **永远不会积压** ✅
**写入稳态**（网络→SRAM→SSD）：
- 网络最高以 100 GB/s 向 SRAM 注入数据
- SSD 以 54 GB/s 消费数据
- 注入速率 > 消费速率 → **存在积压风险** ⚠️
### 5.3 写入场景积压分析（关键风控）

### 7.2 最终结论
> [!IMPORTANT]
> **16MB SRAM 容量充足。** 在所有可实现的稳态工作负载下，DMA 数据暂存区（10MB）的峰值利用率仅为 **35%**（出现在 4K 全随机读场景）。即使在最极端的队列深度要求（每盘 QD=151）下，同时驻留在 SRAM 中的数据也不超过 3.5MB。
>
> **唯一需要注意的是写入反压**：当网络写入速率超过 SSD 吸收能力时（100 GB/s vs 54 GB/s），必须在硬件层面实现 Credit-based 流控或 PFC 反压，防止 SRAM 溢出。这不是容量问题，而是流控设计问题。
### 7.3 安全余量一览
```
16MB SRAM 总量
 ├── 10 MB DMA 缓冲区
 │    ├── 峰值使用: 3.5 MB (4K随机读)
 │    └── 安全余量: 6.5 MB (65% 空闲)
 ├── 4 MB KV 索引缓存
 │    └── 缓存 ~87K 热点条目
 └── 2 MB 队列/描述符
      └── 32K 条目，远超所需
```
